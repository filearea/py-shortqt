# py-shortqt v1.10.0 产品需求设计文档

## 版本目标

**移动端 Web 看板与下单**：在不改动现有 TUI 的前提下，新增一个同进程的轻量 HTTP/WebSocket 服务，将 `TradeState` 实时推送到手机浏览器。PC 端 TUI 和移动端 Web 并存，共享同一套交易引擎和数据源。

**核心能力**：
1. 实时 K 线图渲染（TUI 无法做到的，Web 端原生支持）
2. 移动端监控 + 下单（替代键盘快捷键的触摸交互）

核心使用场景：交易员晚上在床上用手机看 K 线、监控仓位、执行平仓/开仓操作，无需坐在电脑前。

## 一、架构设计

### 1.1 进程内并存模型

```
┌──────────────────── asyncio event loop ────────────────────┐
│                                                             │
│  ┌──────────┐  ┌──────────────┐  ┌──────────────────────┐  │
│  │ 行情 WS   │  │  交易引擎     │  │  Rich TUI (不变)      │  │
│  │ (Binance)│→│  LiveTrader  │→│  src/ui/live_ui.py  │  │
│  └──────────┘  │              │  └──────────────────────┘  │
│                │ TradeState   │                             │
│                │ (共享内存)    │  ┌──────────────────────┐  │
│                └──────┬───────┘  │  Web 服务 (新增)       │  │
│                       │          │  src/web/server.py   │  │
│                       └─────────→│  :8099               │  │
│                                  └──────────┬───────────┘  │
└─────────────────────────────────────────────┼──────────────┘
                                              │ WiFi / 内网
                                    ┌─────────┴─────────┐
                                    │  手机浏览器         │
                                    │  192.168.x:8099    │
                                    └───────────────────┘
```

**关键约束**：
- TUI 代码零改动——它继续刷新它的 Rich Layout
- Web 服务只读 `TradeState`，通过内部消息队列接收操作指令
- 操作来源（键盘 + HTTP 请求）统一路由到交易引擎的同一套接口
- 不存在"谁控制谁"的竞态——键盘事件和 HTTP 请求在同一个 asyncio 事件循环中自然串行化

### 1.2 数据流

```
TradeState (每秒更新)
  │
  ├─→ Rich TUI: 渲染到终端
  │
  └─→ Web 服务:
      ├─ StateSnapshot (JSON) ──WebSocket──→ 手机浏览器
      │    ├ 当前价格、ATR14、振幅
      │    ├ 持仓方向、数量、均价、浮动盈亏
      │    ├ 挂单列表（最多显示 5 条）
      │    └ 风控状态（止损/保底/浮亏保护）
      │
      └─ 操作指令 ←──HTTP POST── 手机浏览器
           ├ 开仓（做多/做空，预设参数）
           ├ 平仓（全部/部分）
           └ 撤单
```

## 二、Web 服务端设计

### 2.1 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| HTTP 框架 | **aiohttp**（新增依赖） | asyncio 原生，可嵌入现有事件循环 |

### 2.4 启动方式

在 `main_live.py` 中，创建 `LiveTrader` 后、进入主循环前，启动 Web 服务：

```python

if config.get('web_ui.enabled', False):
    from src.web.server import start_web_server
    web_runner = await start_web_server(
        trader=trader,
        host=config.get('web_ui.host', '0.0.0.0'),
        port=config.get('web_ui.port', 8099),

    log_manager.info(f"Web UI: http://{local_ip}:{port}?token={token}")
```

### 2.6 K 线数据增强 — 超出本地缓存的历史查询

#### 2.6.1 问题

`/api/klines?limit=100` 只能返回内存中最近 100 根。用户可能需要查看更早的历史（如回溯 3 天前的走势）。

#### 2.6.2 方案

新增历史 K 线查询 API：

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/klines?limit=100&interval=1m` | 最近 N 根（保持兼容） |
| `GET` | `/api/klines/history?from=2026-06-17&to=2026-06-23&interval=1m` | 按日期范围查询 |

**数据来源优先级**：
1. 本地文件 `data/klines/ETHUSDC/日期.jsonl`（按天索引，O(1) 定位）
2. 币安 API（本地无数据的日期范围，按需拉取并缓存）

**前端行为**：K 线图左滑到边界时自动请求更早数据（`from=最早日期&to=当前视口最左`），追加到图表。

**性能约束**：单次最多返回 1440 根（一天），超出需分页。服务端不做 O(n) 遍历——按日期按文件读取。

### 2.7 长时间运行管理

#### 2.7.1 跨日期 ATR14 初始化

**问题**：凌晨启动时当天 K 线不足 14+1 根，ATR14 返回 None，影响止损/止盈计算和盘面显示。

**方案**：`_init_historical_klines()` 启动时，如果当天闭合 K 线 < 15 根，自动从前一天的本地文件（`data/klines/ETHUSDC/昨天.jsonl`）加载尾部 K 线补足。

#### 2.7.2 内存管理

| 数据结构 | 限制策略 | 状态 |
|------|------|------|
| K 线 | `deque(maxlen=200)` | ✅ 有上限 |
| Tick | `PriceRangeTracker` 按时间窗口清理 | ✅ 自清理 |
| `position_history` | **无上限**，长时间运行持续增长 | ⚠️ 需加 `maxlen=500` |
| `error_log` | `maxlen=10` | ✅ |
| 日志文件 | 每日轮转，无过期清理 | ⚠️ 建议 N 天后自动删除（如 30 天） |
| `batch_state` | 分批结束后置 None | ✅ |

#### 2.7.3 定时循环任务清单

| 任务 | 频率 | 触发方式 | 说明 |
|------|------|----------|------|
| WebSocket 行情 | 持续 | `BinanceListener.connect()` | `depth20@100ms` + `bookTicker` + `kline_1m` |
| K 线定时拉取 | 每分钟 | `recorder.start_kline_timer()` | 从 API 获取闭合 K 线 |
| 指标快照 | 每 30 秒 | `metrics_recorder` | 保存到 `data/metrics/` |
| 历史持仓同步 | 每 5 分钟 | `_history_poll` | 兜底轮询 |
| 账户/持仓同步（活跃）| 每 ~5 秒 | `_sync_counter % 100` | 有持仓或挂单时 |
| 账户同步（空闲）| 每 ~15 秒 | `_sync_counter % 300` | 无持仓无挂单时 |
| 持仓同步（空闲）| 每 ~30 秒 | `_sync_counter % 600` | 无持仓无挂单时 |
| Web 状态推送 | 每秒 | `server.py` 定时器 | 给浏览器推 state 快照 |
| Web 订单簿推送 | 每 500ms | `server.py` 定时器 | 给浏览器推 depth 快照 |
| ATR14 24h 百分位 | 每小时 | 服务端后台 | 重算 P50/P75/P95 |

### 2.9 TUI v1.9.0 ATR14 参考值体系

TUI 盘面指标区的 ATR(14) 行增加 24h 滚动百分位评价（与 WebUI 3.3.3.1 一致）：

- 服务端每小时计算过去 24h 所有 1min K 线的 ATR14% 分布，取 P50/P75/P95
- TUI 显示：`ATR(14): 2.14 (0.12%) [P72 🟡]`，圆形字符含义：
  - 🔴 < P50 偏低
  - 🟡 P50–P75 正常
  - 🟢 P75–P95 推荐
  - 🔴 > P95 偏高
- 颜色圆取自已有的 emoji 字符体系，与振幅状态标记风格统一
- 需要对 `VolatilityAnalyzer` 新增 `get_atr14_percentile()` 方法（从 `IndicatorsManager` 获取 24h ATR14% 分布）

### 2.10 K 线图表

#### 2.10.1 技术选型

使用 **TradingView Lightweight Charts**（CDN 引入，~50KB gzip）：

| 特性 | 说明 |
|------|------|
| 蜡烛图渲染 | 原生 OHLC 蜡烛图 + 成交量柱 |
| 触摸交互 | 捏合缩放、滑动平移、十字光标，原生支持移动端 |
| 实时更新 | 支持 `setData()` 增量更新最后一根 K 线 |
| 周期切换 | 前端本地聚合：1m / 5m / 15m / 1h |
| 体积 | 约 50KB gzip，无其他依赖 |
| 许可 | Apache 2.0，免费商用 |

#### 2.10.2 数据来源

系统已有 K 线数据采集（`src/data_collector.py` + `data/klines/`），Web 端需要两个数据通道：

**通道 1：历史 K 线 HTTP API（初始加载）**

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/klines?limit=100&interval=1m` | 返回最近 N 根 1 分钟 K 线 JSON |

返回格式：
```json
[
  {"t": 1781280000, "o": 1741.00, "h": 1743.50, "l": 1740.00, "c": 1742.31, "v": 4521.5},
  ...
]
```

**通道 2：实时 K 线 WebSocket 推送（当前未收盘 K 线）**

每秒推送当前 K 线的 OHLC 更新（最后一根 K 线尚未收盘，随价格实时变化）：

```json
{
  "type": "kline_tick",
  "interval": "1m",
  "t": 1781280060,
  "o": 1742.31,
  "h": 1743.80,
  "l": 1741.00,
  "c": 1743.20,
  "v": 3210.8,
  "is_closed": false
}
```

当前 K 线收盘时 `is_closed` 变为 `true`，前端将其固化，开启下一根。

#### 2.10.3 K 线周期切换

前端本地实现聚合，不请求服务端：

```
1m  → 直接使用原始数据（data_collector 采集的原始粒度）
5m  → 前端按 5 分钟窗口合并 OHLC（取开/最高/最低/收，成交量累加）
15m → 同上，15 分钟窗口
1h  → 同上，1 小时窗口
```

无需服务端参与，前端收到 1m 原始数据后自行聚合。

在前端 K 线图上标记关键价格线（水平虚线）：

| 标记 | 颜色 | 含义 | 优先级 | 最大显示 |
|------|------|------|--------|----------|
| 统一止损线 | 红色实线 | 统一止损触发价 | P1（必须）| 1 |

### 2.12 安全模型

- 启动时生成 32 位随机 hex token，打印在终端日志中
- 所有 API 和 WebSocket 连接必须携带有效 token
- 仅监听内网 IP（`0.0.0.0` 不对外暴露，除非用户做了端口转发）
- 没有 token 的请求返回 403

## 三、移动端 UI 设计

### 3.1 设计原则

1. **拇指操作区**：最频繁的操作（做多/做空/平仓/撤单）放在屏幕底部固定区域
2. **防误触**：仅市价全平使用二次确认弹窗（2 秒倒计时），其余操作即时 Toast 反馈
3. **信息分层**：底部 3 Tab 切换主页面 / 历史覆盖层 / 设置覆盖层，主页面一屏全览
4. **无键盘依赖**：所有操作由触摸按钮完成，零键盘输入
5. **ApeBot 设计系统**：全部 UI 控件使用 ApeBot 原子组件规范，支持深色/浅色双主题
6. **K 线优先**：打开即看 K 线，因为 TUI 做不到这件事是 Web 端的核心差异化价值
7. **红涨绿跌**：遵循中国市场惯例，价格上涨=红色，价格下跌=绿色。语义色与价格色解耦（见 3.9 颜色体系）

##### 3.3.3.2 主动成交比率（Taker Buy/Sell Ratio）— v1.10 新增

**数据源**：币安 `@aggTrade` WebSocket 流，系统内部订阅，由 Web 服务转发给浏览器。

**核心目的**：了解当前盘面上哪一方更主动地在吃单——买方主动（taker buy = 市价买入）vs 卖方主动（taker sell = 市价卖出）。

**计算逻辑**：
- 维护过去 **5 分钟**（300 秒）滚动窗口内的所有 aggTrade 事件
- 每条 aggTrade 的 `m` 字段：`true` = 买方是 maker（卖方主动/卖方taker），`false` = 买方是 taker（买方主动/买方taker）
- 分别累计 taker buy 和 taker sell 的**成交量**（`q` 字段，不是订单笔数）
- 比率 = taker_buy_volume / total_volume × 100% : taker_sell_volume / total_volume × 100%

**UI 显示**：

```
买方:卖方  52.34% : 47.66%
```

- 显示在盘面指标"流动性"的盘口偏向下方
- 买方比例 > 55% → 买方偏绿色（`--color-positive`，买方主动，偏多信号）
- 卖方比例 > 55% → 卖方偏红色（`--color-negative`，卖方主动，偏空信号）
- 45%~55% 之间 → 中性灰色（`--text-secondary`，均衡）
- 字体使用等宽（`--font-mono`），确保百分比对齐

**TUI 显示**（`src/ui/live_ui.py`）：
- 追加在流动性指标行末，不另起一行
- 格式：`主动: 买 52.3%  卖 47.7%`
- 颜色规则同上

**技术实现要点**：
- 在 `BinanceListener.ws_url` 组合流中追加 `/{self.symbol}@aggTrade`
- `src/indicators/taker_ratio.py`（新文件）：维护 deque 滚动窗口，提供 `get_ratio()` 方法
- `IndicatorsManager` 集成，在 `update_agg_trade()` 中更新
- WebSocket 回调新增 `event_type='aggTrade'`，推送 trade 数据给 indicators
- 该指标**仅依赖 WebSocket**，不影响 REST API 消耗
- Web 服务端在 state 快照中新增 `taker_ratio` 字段推送前端

**位置**：持仓卡片在订单簿 & 盘面指标**之前**，K 线图之后，处于页面核心可视区域。

### 3.7.1 Token 错误页

当 token 无效或缺失时，服务端返回独立的错误页面（而非重定向到主页）：

```
┌──────────────────────────┐
│                          │
│          ⚠️             │  ← 大号警告图标
│                          │
│       认证失败           │  ← 一级标题
│   Token 无效或已过期     │  ← 二级说明
│                          │
│   请检查 URL 中的 token  │  ← 操作提示
│   参数，或在 PC 终端      │
│   查看正确的连接地址     │
│                          │
│      [重试] [复制地址]   │  ← 操作按钮
│                          │
└──────────────────────────┘
```

**设计约束**：
- 不渲染任何正常 UI 元素（无顶栏、无 Tab、无 K 线图）
- 只有深色底色 + 品牌色警告图标 + 居中文案
- 重试按钮携带当前 URL 的 token 重新加载
- 复制地址按钮将完整 URL 复制到剪贴板
- 不暴露系统版本号或任何内部信息（安全）
- 使用 `innerHTML` 全量替换 `<body>` 内容，确保没有任何残留元素

### 3.8 响应式适配

- 竖屏（默认）：单列布局，所有内容垂直排列
- 横屏：订单簿与指标并排展示，充分利用宽屏
- 最小支持宽度：320px（iPhone SE 级别）
- K 线图高度固定 200px，折叠后为 0
- 全局隐藏滚动条（`::-webkit-scrollbar { display: none }` + `scrollbar-width: none`）

### 3.9 颜色体系

**核心设计原则**：价格方向色（红涨绿跌）与语义色（绿正常/红异常）解耦，各自独立。

| 变量 | 用途 | 暗色值 | 浅色值 |
|------|------|--------|--------|
| `--color-up` | 价格上涨 / 做多 / 买盘 | `#f6465d` (红) | `#d9304e` |
| `--color-down` | 价格下跌 / 做空 / 卖盘 | `#0ecb81` (绿) | `#0a9e64` |
| `--color-positive` | 正常/成功/胜/在线/推荐 | `#0ecb81` (绿) | `#0a9e64` |
| `--color-negative` | 异常/失败/亏损/断开/风险 | `#f6465d` (红) | `#d9304e` |
| `--color-brand` | 品牌强调/余额/黄金交互 | `#fcd535` (金) | `#c99a0b` |
| `--color-warning` | 警告/注意/中等状态 | `#f0b90b` (黄) | `#c99a0b` |

**使用规则**：
- `.price-up` / `.btn-buy` / `.ob-row.bid` → `--color-up`（红色系）
- `.price-down` / `.btn-sell` / `.ob-row.ask` → `--color-down`（绿色系）
- `.tag-green` / `.status-dot.online` / `.toggle:checked` / `.toast-success` → `--color-positive`（绿色系）
- `.tag-red` / `.status-dot.offline` / `.btn-danger` / `.toast-error` → `--color-negative`（红色系）

**风险提示**：新增组件时需区分使用的是"价格方向色"还是"语义状态色"，混用会导致红绿含义混乱。

### 4.3 设计决策

- 移动端提供**可修改的设置面板**——可查看并修改止盈/止损/分批/系统配置（通过 inline edit），替代 PC 端键盘操作
- 备份管理提供操作按钮（备份/恢复/删除），因为这些操作在手机端也有实际需求
- 移动端覆盖 K线看盘 + 监控 + 交易 + 历史统计 + 设置修改 五个场景
- K 线图是移动端独有的核心竞争力——TUI 完全做不到
- 底部 3 个 Tab 的主页/历史/设置结构，相比 4 Tab 设计减少了切换频率（最常用的看盘+交易在一个页面）
- 仅市价全平使用二次确认弹窗，做多/做空/撤单/提前平仓使用即时 Toast 反馈——平衡防误触和操作效率
- 持仓卡片放在订单簿上方（K线图正下方），作为核心监控区，支持展开/收起切换
- ATR(14) 引入 24h 滚动百分位评价体系（P50-P75 正常/P75-P95 推荐），替代静态阈值判断
- 顶栏交易对名称动态反映当前状态（无持仓/挂单中/持仓盈亏），减少视线移动
- 脱敏模式下点击余额可重置脱敏基数（等价 TUI 的 R 键）
- 流动性指标文本固定宽度右对齐，防止"买盘占优"/"均衡"长度不一致导致的布局闪烁
- 设置面板交易 tab 包含实时盈亏预览，与 TUI `_render_trading_tab_lines()` 预览段功能对齐
- 主题切换从顶栏移至设置-系统子标签（减少顶栏视觉噪音）
- 颜色体系解耦：价格方向色与语义状态色分离，新增组件时必须遵循（见 3.9）

### 5.2 K 线实时 Tick（每秒推送）

```json
{
  "type": "kline_tick",
  "interval": "1m",
  "t": 1781280060,
  "o": 1742.31,
  "h": 1743.80,
  "l": 1741.00,
  "c": 1743.20,
  "v": 3210.8,
  "is_closed": false
}
```

`is_closed = true` 表示该 K 线已收盘，前端固化并开始下一根。1 分钟 K 线收盘时立即推送，其他周期由前端自行判断。

### 5.3 增量事件

除了全量快照和 K 线 tick，特定事件触发时立即推送增量消息（不等 1 秒间隔）：

```json
{"type": "event", "event": "order_filled", "detail": "批2 成交 @2998.00"}
{"type": "event", "event": "tp_hit", "detail": "批1 止盈 @3002.00"}
{"type": "event", "event": "sl_triggered", "detail": "止损触发 @2993.00"}
{"type": "event", "event": "connection_lost", "detail": "WebSocket 断开"}
```

前端收到事件后播放简短提示音（Web Audio API 生成，无需音频文件）并短暂振动（`navigator.vibrate`）。

### 5.4 订单簿推送

**订单簿由系统转发，浏览器绝不直连币安。** 服务端通过行情 WebSocket（`depth20@100ms`）实时接收，每 500ms 节流推送给浏览器：

```json
{
  "type": "depth",
  "ts": 1781280000,
  "bids": [["1742.31", "12.50"], ["1742.28", "8.30"], ...],
  "asks": [["1742.32", "10.30"], ["1742.35", "7.80"], ...],
  "spread": 0.01,
  "spread_pct": 0.0006,
  "pressure": "buy",
  "pressure_pct": 54.8
}
```

- `bids`/`asks`：各 8 档（与 TUI 一致），价格和数量均为字符串
- `spread`：买卖一价差（USDC）
- `spread_pct`：价差百分比
- `pressure`：盘口偏向 — `"buy"`（买盘占优）/ `"sell"`（卖盘占优）/ `"balanced"`（均衡）
- `pressure_pct`：买盘深度占买卖总深度的百分比
- 推送频率 500ms（比内部 `depth20@100ms` 低，节省移动端带宽）
- 前端用此数据渲染订单簿（替代 HTTP 轮询）

## 七、兼容性与约束

### 7.1 浏览器兼容

| 浏览器 | 最低版本 | 说明 |
|--------|---------|------|
| Chrome Android | 90+ | 完全支持 |
| Safari iOS | 15+ | 完全支持 |
| Firefox Android | 90+ | 完全支持 |
| 微信内置浏览器 | — | 可能有限制，建议用系统浏览器打开 |

### 7.2 性能约束

- WebSocket 推送频率：1Hz（每秒一次全量快照）
- 增量事件：即时推送，无节流
- 前端渲染：纯 DOM 操作，不依赖虚拟 DOM 框架
- 内存占用：Web 服务端约 5-10MB 额外内存
- CPU 开销：JSON 序列化 TradeState 约 0.1ms

### 7.3 安全约束

- Web 服务仅在内网 IP 上监听（`0.0.0.0` 不对外暴露）
- 所有请求需要 token 认证
- 生产环境不建议通过公网直接暴露，如需远程访问应使用 VPN 或 SSH 隧道

### 7.4 已知限制

| 限制 | 影响 | 缓解措施 |
|------|------|---------|
| 设置可修改 | 移动端修改的参数影响 PC TUI 行为 | 所有参数修改即时生效（与 TUI S 键保存等价） |
| 无历史回看 | 仅显示近期历史持仓列表 | 需要完整复盘时回 PC |
| 仅一个并发移动端 | 多人同时连接会看到相同数据但操作冲突 | 设计上仅支持单人使用（个人交易工具） |
| 无推送通知 | 手机关屏后不会收到成交提醒 | 建议不锁屏或将浏览器保持前台 |

## 九、技术评审

### 9.1 架构评审

**✅ 优点**：
- 进程内并存模型正确：asyncio 事件循环天然串行化，避免了"PC和手机同时操作"的竞态问题
- 原计划 TUI 零改动，技术评审后确认少量必要改动（代理配置、ATR14 百分位、内存管理），改动范围可控
- `TradeState` 作为共享内存读源，Web 服务只读不写，不会引入并发 bug

**⚠️ 关注点**：

1. **aiohttp 与现有 ws 库的兼容性**：项目已有的 `src/websocket.py` 使用 `websockets` 库。aiohttp 使用自己的事件循环管理，两者在同一个 asyncio loop 中运行通常没问题，但需注意 aiohttp 的 `AppRunner` 生命周期管理——必须在 `main_live.py` 的 `finally` 块中正确 cleanup，否则可能导致优雅退出失败。

2. **JSON 序列化开销**：每秒序列化完整 `TradeState` 约 2-5KB JSON。在低端设备（树莓派）上应该没问题，但如果未来增加更多字段，建议对 `pending_orders` 和 `tp_orders` 做增量推送而非每次全量。

3. **Token 在 URL query string 中**：浏览器历史记录和日志会保留 token。建议同时支持 `Authorization: Bearer <token>` header 方式，前端优先使用 header。

### 9.2 前端评审

**✅ 优点**：
- 纯 HTML/CSS/JS 单文件方案零依赖，部署简单，符合项目定位
- ApeBot 设计系统 CSS 变量体系完整，主题切换和颜色管理清晰
- 颜色体系解耦（价格方向色 vs 语义状态色）是正确的设计决策，避免了"红涨绿跌"对 UI 控件的污染

**⚠️ 关注点**：

1. **TradingView Lightweight Charts CDN**：`unpkg.com` 在国内访问可能不稳定。建议本地托管 `tv-charts.umd.js`（约 50KB gzip），已在文件清单中预留。首次加载时如 CDN 不可用需有 fallback 提示。

2. **1Hz 轮询下的 DOM 操作频率**：每秒更新价格、指标、订单簿、持仓盈亏。当前使用 `innerHTML` 直接替换——对 1660 行规模的页面可以接受，但如果日志/订单簿数据量增长，建议切换到增量 DOM 更新（只更新变化的节点）。`innerHTML` 还有一个隐患：会销毁已有的事件监听器。

3. **K 线数据聚合**：前端本地做 1m→5m/15m/1h 聚合是正确的设计（减少服务端计算和传输）。但注意 `lightweight-charts` 的 `setData` 是全量替换——周期切换时需要重新计算全部数据点，100 根 K 线的 O(n) 聚合可以接受。

4. **inline edit 安全性**：当前原型中 inline edit 的值直接用于 `innerHTML`，没有 XSS 防护。生产代码中必须对用户输入做 `textContent` 赋值或 sanitize。特别是 IP 地址字段使用了 text 类型——需校验格式。

5. **条件显隐的状态一致性**：设置面板的条件字段显隐依赖 `onchange` 事件。如果通过程序化方式（如加载配置）修改 select 值，需额外触发 change 事件或手动调用对应的 handler。

### 9.3 设置面板评审

**✅ 优点**：
- 与 TUI `settings_ui.py` 的字段定义完整对齐，不会出现"PC 能配但手机看不到"的情况
- 实时计算预览（盈亏结果）替代了之前的杠杆预览，与 TUI `_render_trading_tab_lines()` 对齐
- inline edit 交互符合移动端操作习惯

**⚠️ 关注点**：

1. **设置保存的竞态**：移动端修改参数 → 保存 → HTTP POST 到服务端 → 写入 config → TUI 设置面板如正在编辑同一字段可能出现覆盖。建议服务端对 config 写入加简单锁，或前端保存时携带 `expected_version` 做乐观锁。

2. **代理配置**：这是 WebUI 原型中新增的配置节（`proxy.enabled` / `proxy.host` / `proxy.port`），TUI 设置面板（`settings_ui.py` tabs[4]）目前没有对应字段。实际开发时需决定：要么在 TUI 系统设置 tab 中新增代理字段，要么代理配置仅 WebUI 可配。建议前者——如果用户通过代理连 Binance，PC 端也应该能配置。

3. **时区输入**：IANA 时区标识符是自由文本输入，服务端必须用 `pytz` 或 `zoneinfo` 做校验。无效时区应返回友好错误而非静默 fallback 到 UTC。建议前端提供常用时区 datalist 辅助输入。

4. **订单超时和分批模式互斥**：v1.9.0 中分批模式启用时订单超时禁用。当前设置面板两个字段同时可见，未做互斥提示。建议在分批 tab 和交易 tab 超时字段旁加提示文字。

### 9.4 WebSocket 协议评审

**✅ 优点**：
- 全量快照 + 增量事件的推送模型正确，既保证了状态一致性又提供了即时反馈
- K 线 tick 独立于状态快照的设计合理，K 线更新频率可以独立调整
- 新增订单簿 depth 推送频道（5.4），解决评审中发现的订单簿缺失问题

**⚠️ 关注点**：

1. **`last_position` 数据来源**：当前设计中 `last_position` 包含 `fee`、`funding`、`net_pnl` 等字段。这些数据需要从历史持仓记录中提取。确保服务端在构建 `StateSnapshot` 时能高效取到最近一条记录（O(1) 而非遍历）。

2. **`pending_orders` 和 `tp_orders` 的排序**：WebSocket 推送的数组顺序应与前端显示顺序一致。做多/做空的不同排序规则（做多止盈低→高、做空止盈高→低）应由服务端排序后推送，减轻前端逻辑。

3. **重连后的状态同步**：前端 WS 断开重连后，应能通过 `/api/state` HTTP 接口获取完整当前状态作为初始值，避免短暂的"无数据"闪烁。

4. **订单簿推送由系统转发**：浏览器绝不直连币安。服务端已有 `depth20@100ms` 行情流，内部 100ms 更新供 TUI 使用，每 500ms 节流推送给浏览器（节省移动端带宽）。**v1 设计文档初版遗漏了此频道**——必须在 state 快照之外独立推送，因为订单簿变化频率远高于持仓状态。

### 9.5 安全评审

1. **Token 管理**：32 位 hex token 足够安全（2^128 空间）。但需确保 token 在服务端是常量（启动时生成一次），不要每次请求重新生成。Token 不应写入日志文件。

2. **CORS**：如果前端从 `file://` 协议打开（本地 HTML 文件），浏览器会阻止跨域请求。建议服务端设置适当的 CORS 头或强制通过 HTTP 服务访问。

3. **速率限制**：HTTP API（开仓/平仓/撤单）应加基本的速率限制，防止误触或恶意频繁调用。建议每 500ms 最多 1 次操作请求。

4. **输入校验**：所有 HTTP POST 参数必须在服务端校验——包括 side（仅 LONG/SHORT）、percent（0-100）、price（正数且符合 tickSize）等。不要信任前端传来的任何值。

### 9.6 兼容性评审

1. **Safari iOS 的 100vh 问题**：iOS Safari 的 `100vh` 包含底部工具栏高度，可能导致底部 Tab 被遮挡。建议使用 `dvh` 单位或 JS 动态计算高度。

2. **`navigator.vibrate` 兼容性**：iOS Safari 不支持 Vibration API。音效提示在 iOS 上需要通过用户手势触发（Web Audio API 的 autoplay 限制）。建议优先使用 AudioContext，在首次用户点击时 `resume()`。

3. **`scrollbar-width: none` 兼容性**：Firefox 支持，Chrome 121+ 支持，旧版 Chrome 需要 `::-webkit-scrollbar`。当前两套都写了，覆盖良好。

## 十四、v1.10.0 发布后修复（2026-06-28）

### 14.1 K 线数据质量防护（recorder.py + main_live.py）

**问题**：币安 API 对刚闭合的 K 线可能返回未最终确定的数据（`buy_turnover=0`），导致本地文件存储了不完整的 K 线。每天 1440 根中有约 999 根存在此问题。

**修复**：
- `recorder._is_kline_finalized()`: 判断 K 线闭合是否超过 10 秒（数据已最终确定）
- `_fetch_and_save_kline()`: 拉取窗口从 5 扩大到 10 根；对齐从第 2 秒推迟到第 10 秒；跳过 `volume>0` 但 `buy_turnover<=0` 的未最终确定数据
- `_correct_recent_dirty_klines()`: 运行时自愈——检查文件最后 15 条，发现脏数据后从 API 回补并重写文件
- `_init_historical_klines()`: 从 API 回写本地文件时同样跳过脏数据
- `_on_kline_closed` 回调：K 线收盘时间步触发 `track_atr14_percentile()`

### 14.2 WebSocket 双流架构（websocket.py）

**问题**：组合流中的 `@aggTrade` 和 `@kline_1m` 在代理环境下被阻断，导致主动成交比率无数据、K 线无法实时合成。

**修复**：
- 拆分为双流并行连接：组合流（`bookTicker + depth20@100ms`）+ 独立 `@trade` 流（逐笔成交）
- `@trade` 逐笔成交在本地实时合成 1m OHLCV K 线，替代被阻断的 `@kline_1m`
- 异常价保护：偏离参考价 >10% 的成交不参与合成
- `aggTrade` 和 `trade` 事件类型统一处理，`TakerRatio` 新增 `trade_count`/`buy_count`/`sell_count`/`last_update` 字段

### 14.3 H/L 价格范围数据源修正（server.py）

**问题**：Web UI 头栏的"近X分钟最高/最低价"使用 `server.py` 内临时创建的 `_price_ticks` deque，启动后从零开始积累，无历史数据，显示偏差严重。

**修复**：移除临时 deque，改为直接读取已有的 `indicators.price_range`（`PriceRangeTracker`），该 tracker 在启动时通过 `seed_from_klines()` 回填历史 K 线高/低价，运行时由 `indicators.update_tick()` 实时更新。

### 14.7 14天K线回填修复（data_collector.py）

**问题**：启动时 14 天历史 K 线回填完全失败，本地只有当天数据。

**根因**：
- `is_complete` 变量只在 `file_path.exists()` 分支内定义（line 226），当文件不存在时 NameError，导致 `fetch_klines()` 从不执行
- K 线字段索引错误：`turnover` 用了 k[6]（close_time 时间戳）而非 k[7]（quote volume），`buy_volume` 用了 k[10] 而非 k[9]，`buy_turnover` 用了 k[11] 而非 k[10]

**修复**：`is_complete = False` 提前初始化；修正全部 3 个字段索引。

### 14.8 Web UI 下单接口修复 — 复用 TUI 操作方法（server.py + main_live.py）

**问题**：4 个下单/撤单 API 全部返回"网络错误"（500）。根因是 handler 调用了旧 `TradeState` 模拟类的方法（`long()`/`short()`/`close_all_positions()`/`cancel_all_orders()`），这些方法在 `LiveTrader` 上不存在。

**修复**：改为调用正确的 `LiveTrader` 方法，并确保与 TUI 键盘快捷键走完全相同的逻辑路径：

| Web 操作 | 调用方法 | 对应 TUI |
|---------|---------|---------|
| 做多 | `app.place_order('LONG')` | ↑ 键 |
| 做空 | `app.place_order('SHORT')` | ↓ 键 |
| 全部平仓（市价） | `trader.close_position_market()` | Z 键 |
| 提前平仓 | `app.close_position_early()` | → 键 |
| 撤单 | `app.cancel_order()` | ← 键 |

**关键设计**：`WebServer` 新增 `app` 引用（`LiveTradingBot` 实例），Web handler 通过 `self.app.xxx()` 调用 App 层的操作方法，而非直接调 `trader` 方法。这样任何未来在 App 层增加的逻辑（如 `cancel_order()` 中的分批模式检查）都会被 Web 侧自动继承。

### 14.10 平仓自动刷新机制强化（live.py + index.html）

**问题**：平仓后「上一笔持仓」卡片不自动更新，需手动到历史页刷新。根因是 `_refresh_history_delayed` 5s 延迟 + 3 次重试（总窗口 14s）不足以等 Binance 同步 trade 记录。

**修复**：
- `_refresh_history_delayed`: 延迟 6s，重试 5 次 × 间隔 4s = 总窗口 26s；增加成功/失败日志
- `fetch_position_history`: 返回 bool 表示成功/失败，仅成功时推送 `history_updated`
- 前端平仓事件 fallback: `fetchState()` 延迟调用从 6s/12s 改为 8s/16s/24s，对齐后端重试窗口
- `history_updated` WebSocket 推送 → `fetchState()` → `updateUI()` → `updatePrevPosition()` 全链路打通

### 14.11 K 线缺口检测与三层兜底回写（server.py）

**问题**：前端无浏览器连接期间，内存 K 线队列出现缺口（recorder 线程在后台正常写入但前端 `klineData` 不更新）。原回退逻辑仅按数量判断，无法检测队列中间缺口。

**修复**：
- `_handle_klines`: 改为时间连续性检测——相邻 K 线时间差 > 60s 即判定为缺口
- `_fill_kline_gaps`: 缺口填补链 → 内存 deque → 本地 JSONL 文件 → 币安 API
- `_writeback_klines`: 从文件/API 补回的 K 线写回内存 deque + JSONL 文件
- WebSocket 重连 `onopen` 触发 `loadKlines()` 刷新 K 线数据
- `_lastTickTime` / `_lastTickClose` 在 `loadKlines` 后同步，防止误判缺口
