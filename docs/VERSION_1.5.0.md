# py-shortqt v1.5.0 - 移动止损与浮亏保护

> **发布日期**: 2026-04-01（计划）  
> **核心功能**: 浮盈移动止损 + 浮亏保护止损  
> **版本定位**: py-shortqt 封存前最后一个功能版本

---

## 📊 版本概览

v1.5.0 为交易系统增加两种智能止损机制：

1. **浮盈移动止损（Trailing Stop）** - 盈利时自动上移止损，锁定利润
2. **浮亏保护止损（Loss Protection）** - 亏损时自动下移止盈，保本出场

**核心价值**:
- 📈 浮盈时最大化利润，避免过早平仓
- 📉 浮亏时及时止损，避免深度套牢
- ⚙️ 两种机制独立配置，可同时开启

## 🆕 核心功能

### 1. 浮盈移动止损（Trailing Stop）

#### 1.1 功能描述

当持仓产生浮盈后，系统自动在开仓价与目标止盈价之间设置多个移动止损档位，随着价格上涨逐步上移止损位，锁定既得利润。

#### 1.2 配置参数

| 参数 | 说明 | 默认值 | 范围 |
|------|------|--------|------|
| `enabled` | 是否启用 | `false` | - |
| `grid_count` | 移动止损格数 | `5` | 3-10 |

**配置位置**: `config/runtime.json`

```json
{
  "trailing_stop": {
    "enabled": true,
    "grid_count": 5
  }
}
```

#### 1.3 算法逻辑

**1. 价格区间均分**

开仓后，系统自动在 **开仓价 → 目标止盈价** 之间均分 N 格：

```
示例配置:
- 开仓价: 2100
- 目标止盈: 2115
- 移动止损格数: 5 格

均分后价格点:
┌─────────────────────────────────┐
│ 2115  ← 目标止盈                 │
│ 2112  ← 第 4 格（移动止损位 4）    │
│ 2109  ← 第 3 格（移动止损位 3）    │
│ 2106  ← 第 2 格（移动止损位 2）    │
│ 2103  ← 第 1 格（移动止损位 1）    │
│ 2100  ← 开仓价                   │
└─────────────────────────────────┘
每格间距 = (2115 - 2100) / 5 = 3 点
```

**2. 触发逻辑**

当最新价超过第 N 格时，设置条件止损单在第 N-1 格触发：

```
多单示例（开仓价 2100）:

当前价 > 2103 → 设置止损单 @ 2100（开仓价保本）
当前价 > 2106 → 设置止损单 @ 2103（锁定 3 点利润）
当前价 > 2109 → 设置止损单 @ 2106（锁定 6 点利润）
当前价 > 2112 → 设置止损单 @ 2109（锁定 9 点利润）
当前价 > 2115 → 设置止损单 @ 2112（锁定 12 点利润）
```

**3. 订单管理**

- **订单类型**: 币安 Algo Order `type='STOP'`
- **执行方式**: `priceMatch='QUEUE'`（同向价一，Maker 成交）
- **最大单数**: 8 个（币安限制 10 个 - 开仓止盈止损 2 个）

**格数限制**:
- 最小：3 格
- 最大：10 格（但最多同时存在 8 个移动止损单）
- 如果配置 > 8 格，采用滚动策略（见下文）

#### 1.4 滚动策略（格数 > 8 时）

当用户配置格数超过 8 格时，系统自动滚动剔除较早的止损订单：

```
配置：10 格
币安限制：最多 8 个条件单

触发流程:
1. 价格超过第 1 格 → 创建止损单 @ 第 0 格（剩余可用：7）
2. 价格超过第 2 格 → 创建止损单 @ 第 1 格（剩余可用：6）
...
8. 价格超过第 8 格 → 创建止损单 @ 第 7 格（剩余可用：0）
9. 价格超过第 9 格 → 撤销第 0 格止损单，创建止损单 @ 第 8 格
10. 价格超过第 10 格 → 撤销第 1 格止损单，创建止损单 @ 第 9 格
```

**保留规则**:
- ✅ 必须保留：开仓时的原始止损单（保底止损）
- ✅ 必须保留：开仓时的原始止盈单（目标止盈）
- 🔄 可滚动：移动止损单（按触发价格从旧到新滚动）

#### 1.5 空单适配

空单逻辑与多单对称：

```
空单示例（开仓价 2100，目标止盈 2085）:

均分 5 格:
- 2100  ← 开仓价
- 2097  ← 第 1 格
- 2094  ← 第 2 格
- 2091  ← 第 3 格
- 2088  ← 第 4 格
- 2085  ← 目标止盈

触发逻辑:
当前价 < 2097 → 设置止损单 @ 2100（开仓价保本）
当前价 < 2094 → 设置止损单 @ 2097（锁定 3 点利润）
当前价 < 2091 → 设置止损单 @ 2094（锁定 6 点利润）
...
```

### 2. 浮亏保护止损（Loss Protection）

#### 2.1 功能描述

开仓后经过指定时间，如果持仓仍处于浮亏状态，系统自动将止盈单下移到开仓价，确保后续价格反弹时能够保本出场，避免深度套牢。

#### 2.2 配置参数

| 参数 | 说明 | 默认值 | 范围 |
|------|------|--------|------|
| `enabled` | 是否启用 | `false` | - |
| `trigger_minutes` | 触发检测时间（分钟） | `5` | 1-60 |

**配置位置**: `config/runtime.json`

```json
{
  "loss_protection": {
    "enabled": true,
    "trigger_minutes": 5
  }
}
```

#### 2.3 算法逻辑

**1. 时间触发**

```
开仓时间：10:00:00
配置时间：5 分钟
检测时间：10:05:00

→ 触发浮亏检测
```

**2. 状态判断**

```
检测时刻持仓状态:
┌─────────────────────────────────┐
│ 浮盈（未实现 PnL > 0）           │
│ → 不做操作，保持原止盈单         │
├─────────────────────────────────┤
│ 浮亏（未实现 PnL < 0）           │
│ → 执行保护动作                   │
└─────────────────────────────────┘
```

**3. 保护动作**

```
浮亏时执行:
1. 撤销原止盈单（目标止盈价）
2. 新建止盈单 @ 开仓价（保本出场）

效果:
- 如果价格反弹到开仓价 → 保本平仓
- 如果价格继续下跌 → 触发原始止损单
```

#### 2.4 示例场景

**场景 A：浮亏后反弹保本**
```
10:00  开仓 @ 2100，止盈 @ 2115，止损 @ 2085
10:05  价格跌至 2095（浮亏 -5 点）
       → 触发浮亏保护
       → 止盈单下移至 2100（开仓价）
10:15  价格反弹至 2100
       → 止盈单成交，保本出场
```

**场景 B：浮亏后继续下跌**
```
10:00  开仓 @ 2100，止盈 @ 2115，止损 @ 2085
10:05  价格跌至 2095（浮亏 -5 点）
       → 触发浮亏保护
       → 止盈单下移至 2100
10:20  价格继续跌至 2085
       → 原始止损单触发，止损出场
```

**场景 C：浮盈不触发**
```
10:00  开仓 @ 2100，止盈 @ 2115，止损 @ 2085
10:05  价格涨至 2105（浮盈 +5 点）
       → 不触发浮亏保护
       → 保持原止盈单 @ 2115
```

#### 2.5 空单适配

空单逻辑对称：

```
空单示例:
10:00  开空 @ 2100，止盈 @ 2085，止损 @ 2115
10:05  价格涨至 2105（浮亏 -5 点）
       → 触发浮亏保护
       → 止盈单上移至 2100（开仓价）
```

## ⚠️ 注意事项

### 1. 币安订单限制

| 限制类型 | 数值 | 说明 |
|----------|------|------|
| 最大限价单数 | 200 | 开仓 + 止盈 + 止损都占用 |
| 最大条件单数 | 10 | 移动止损最多用 8 个 |
| 最小名义价值 | ~5 USDT | 每笔订单必须满足 |

### 2. API 限流

| 接口 | 限流 | 应对策略 |
|------|------|----------|
| 创建条件单 | 10 次/秒 | 批量操作时注意间隔 |
| 撤销条件单 | 10 次/秒 | 滚动策略时控制频率 |
| 查询条件单 | 20 次/秒 | 缓存结果，减少查询 |

### 3. 网络异常处理

- 创建订单失败 → 记录日志，下次价格更新时重试
- 撤销订单失败 → 记录日志，避免重复创建
- WebSocket 断连 → 暂停移动止损，恢复后重新同步

### 4. 并发控制

- 条件单操作需要原子性（创建/撤销）
- 使用队列串行化处理
- 避免同时操作同一订单
