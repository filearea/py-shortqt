# py-shortqt v1.9.0 产品需求设计文档

## 版本目标

1. **分批建仓**：解决资金量增大后，单一限价单挂在 BBO 无法吃下足够流动性的问题。引入分批建仓模式，将一笔大单拆为 X 笔阶梯分布的限价单，分批成交，独立止盈，统一止损。
2. **WebSocket 可靠性修复**：解决用户数据流 WS 表面连接正常但无数据流入（僵尸连接）的问题，增加应用层心跳、Listen Key 自动轮换、REST 轮询兜底。

---

## 一、功能概述

### 1.1 核心流程

```
用户按↑/↓做多/做空
    │
    ▼
系统计算总名义价值 = 可用余额 × 实际杠杆
    │
    ▼
按配置拆分 X 笔，按阶梯规则计算每笔价格和数量
    │
    ▼
通过 batchOrders 接口（每 ≤5 笔一组）批量挂出 X 笔限价单
   — 优先 GTX（Post-Only Maker）
   — GTX 被拒 → 降级 GTC 限价单补挂
    │
    ▼
等待各笔陆续成交 ──→ 每笔成交后：
│                       ① 为该笔挂独立止盈限价单
│                       ② 计算所有已成交批次加权均价
│                       ③ 撤销旧的 SL/SM（≥3 秒节流）
│                       ④ 按新均价挂统一 SL/SM
│
    ▼
全部成交 → 进入"完全持仓"状态，等待止盈/止损/保底触发
```

### 1.2 轮次结束条件

分批建仓的一个"持仓轮次"在以下任一条件满足时结束：

1. **全部止盈成交**：至少有一笔止盈已挂出，且所有止盈单都已成交 → 立即撤销所有剩余挂单（含未成交批次）、止损单、保底单，轮次结束
2. **止损触发**：统一止损（币安 STOP_MARKET）被触发 → 由币安自动市价执行，系统清理所有剩余订单
3. **保底止损触发**：统一保底（币安 STOP_MARKET）被触发 → 同上
4. **Z 键市价全平**：用户手动全平 → 撤销一切

**关键逻辑**：只要所有已挂出的止盈单全部成交，即使还有未成交的挂单批次，也视为本轮结束。场景：部分批次成交后价格朝盈利方向飞奔，吃掉所有止盈单，剩余未成交批次不再有意义。

### 1.3 模式开关

- 配置项 `batch_mode.enabled`，默认 `false`
- 关闭时，系统行为与 v1.8.x 完全一致（单笔开仓）
- 开启后，方向键开仓自动走分批流程

## 二、配置参数

### 2.1 配置项清单

| 键 | 类型 | 默认值 | 说明 |
|------|------|---------|------|
| `batch_mode.enabled` | bool | `false` | 是否启用分批模式 |
| `batch_mode.count` | int | `5` | 拆单笔数 X（2~50） |
| `batch_mode.distribution` | select | `equal` | 数量分配方式 |
| `batch_mode.ladder_mode` | select | `fixed` | 价格阶梯方式 |
| `batch_mode.ladder_min` | decimal | `1.00` | 阶梯最近端（最小偏离 BBO） |
| `batch_mode.ladder_max` | decimal | `10.00` | 阶梯最远端（最大偏离 BBO） |

### 2.2 数量分配方式 (distribution)

| 值 | 标签 | 逻辑 |
|------|------|------|
| `equal` | 均分 | 每笔 ETH 数量 = 总量 / X |
| `increase` | 递增 | 第一笔最小，逐笔增大，最后一笔最大 |
| `decrease` | 递减 | 第一笔最大，逐笔缩小，最后一笔最小 |
| `random` | 随机 | 每笔在均分值的 ±5% 范围内随机浮动，总和等于目标总量 |

### 2.3 价格阶梯方式 (ladder_mode)

| 值 | 标签 | 逻辑 |
|------|------|------|
| `fixed` | 固定点数 | 步距单位为 USDC 点数 |
| `percentage` | 百分比 | 步距单位为当前价 × 百分比 |
| `atr14` | ATR14系数 | 步距单位为 ATR(14) |

**阶梯范围**：配置 `ladder_min`（最近端，最小偏离 BBO）和 `ladder_max`（最远端，最大偏离 BBO），系统按 X 笔数量自动计算每档步距。

- **做多(LONG)**：第 1 笔买一价 − ladder_min，第 X 笔买一价 − ladder_max，中间等距插值
- **做空(SHORT)**：第 1 笔卖一价 + ladder_min，第 X 笔卖一价 + ladder_max，中间等距插值

插值公式：`步距 = (ladder_max − ladder_min) / (X − 1)`

**精度取舍**：每笔价格按 ETHUSDC tickSize（0.01）截断；最后一笔价格强制对齐到 ladder_max 计算值，避免累积误差。

### 2.4 Maker 优先策略

**开仓挂单**：
- 全部批次以 `timeInForce: "GTX"` 提交 batchOrders
- GTX 被拒（错误码 `-5022`）：**不自动补单**，记录失败数，TUI 日志打印 "分批开仓 — X/Y 笔 GTX 挂单成功，Z 笔失败等待手动补单"
- 失败批次不记录到 batch_state（未成功挂出的不追踪）

**止盈挂单**：
- 止盈单也用 `timeInForce: "GTX"`
- GTX 被拒 → 自动降级为 GTC 限价单补挂（BBO 同侧最优价），因为止盈是时间敏感的
- 补挂仍失败 → 日志记录，该笔止盈标记为异常

**恢复止盈单**（提前平仓撤销后）：
- 从 tp_backup 恢复时，同样用 GTX 挂出
- GTX 被拒 → 降级 GTC BBO 补挂

### 2.5 Binance 约束保护

- 每笔名义价值 ≥ 20 USDC（币安最小名义价值）
- 每笔 ETH 数量 ≥ 0.001（ETHUSDC 最小数量精度）
- 如果计算出的某笔不满足约束，自动减少批次数或提示用户降低杠杆

## 三、批次状态机与核心生命周期

### 3.1 轮次级状态

```
                    ┌─────────┐
                    │  IDLE   │ ← 无持仓、无挂单
                    └────┬────┘
                     ↑/↓ 开仓
                         │
                    ┌────▼─────────┐
                    │ PENDING_ONLY │ ← X 笔已挂出，无一成交
                    └────┬─────────┘
                    首笔成交
                         │
          ┌──────────────┼──────────────┐
          │              │              │
    后续批次成交     → 提前平仓     Z / SL触发
          │              │              │
          ▼              ▼              ▼
    ┌──────────┐  ┌────────────┐   ┌──────┐
    │ 部分持仓  │  │EARLY_CLOSE │   │IDLE  │
    │(含已成交 │  │(提前平仓中) │   └──────┘
    │ +挂单   │  └─────┬──────┘
    │ +止盈   │    ← 撤销
    │ +SL/SM) │        │
    └────┬─────┘   ┌────▼───────┐
         │         │ 恢复到部分 │
    最后挂单成交    │ 持仓状态   │
         │         └────────────┘
         ▼
    ┌──────────┐
    │ 完全持仓  │ ← 全部批次成交，所有止盈挂出
    └────┬─────┘
    止盈逐一触发
         │
         ▼
    ┌──────────┐
    │ 部分止盈  │ ← 部分止盈已成交，部分仍在
    └────┬─────┘
    全部止盈成交
         │
         ▼
    ┌──────┐
    │ IDLE │ ← 轮次结束，清理所有剩余订单
    └──────┘
```

### 3.2 单批次状态

```
pending ──→ filled ──→ tp_placed ──→ tp_closed
  │            │                        │
  │            ├── failed (EXPIRED)      │
  │            │                        │
  └── cancelled（←撤单）                └── closed（止损/SL触发）
```

- `pending`: 挂单中，等待成交
- `filled`: 已成交，尚未挂止盈单（瞬态）
- `tp_placed`: 已成交且止盈单已挂出
- `tp_closed`: 该批次止盈成交，仓位已平
- `closed`: 该批次通过止损/保底/市价全平方式关闭
- `cancelled`: 挂单被撤销（从未成交）
- `failed`: 订单过期未成交（EXPIRED），不产生止盈单

### 3.3 ← 键的上下文语义

← 键的行为取决于当前状态：

| 当前状态 | ← 键行为 |
|---------|---------|
| PENDING_ONLY（仅有挂单，无成交） | 撤销所有未成交批次 → IDLE |
| PARTIAL_FILLED（有已成交 + 有挂单） | 撤销所有未成交批次，已成交部分和止盈单保留 |
| ALL_FILLED / PARTIAL_TP | 无效操作，提示"已全部成交" |
| **EARLY_CLOSE（提前平仓中）** | **撤销提前平仓单，恢复备份的止盈单，回到 PARTIAL_FILLED** |

**关键区分**：EARLY_CLOSE 状态下 ← 撤销的是提前平仓单，不是未成交批次。此时未成交批次仍挂在盘口。

## 四、逐笔成交逻辑

### 4.1 成交检测

- 通过 WebSocket `ORDER_TRADE_UPDATE` 事件按 `orderId` 匹配批次
- 通过关键价格穿透检测（`_check_key_prices`）兜底
- 维护 `orderId → 批次索引` 映射表

### 4.2 成交后处理流程

```
批次k成交（数量 size_k，价格 price_k）
    │
    ├─ ① 标记批次k为"filled" → "tp_placed"，记录入场价 price_k
    │
    ├─ ② 为该批次挂独立止盈限价单
    │     止盈价 = price_k + 止盈点数/百分比/ATR（按当前止盈配置）
    │     止盈数量 = size_k
    │
    ├─ ③ 计算所有已成交批次加权均价
    │     加权均价 = Σ(price_i × size_i) / Σ(size_i)
    │
    ├─ ④ 撤销旧的 SL/SM 算法订单（**合并节流：≥3 秒间隔**）
    │
    └─ ⑤ 按加权均价 + 当前止损/保底配置，重新挂 SL/SM
           止损/保底数量 = 已成交总量
```

**并发成交合并**：若 WebSocket 在 3 秒内连续推送多笔成交，只触发一次 SL/SM 撤销+重挂（取最后一批成交后的均价），避免频繁操作触发币安限频。

**均价计算规则**：仅在开仓挂单成交时触发。使用所有未止盈（`filled` + `tp_placed`）批次的加权均价。止盈成交时只减少剩余活跃仓位，不触发均价重算。

### 4.3 全部成交后

- 状态切换为"完全持仓"（ALL_FILLED）
- 等待各独立止盈单或统一止损/保底触发
- 不再有未成交的挂单

### 4.4 订单成交失败（保证金不足）

**场景**：批次挂单已在盘口，但价格到达时因保证金不足等原因被币安拒绝成交。

**检测方式**：
- 监听 WebSocket `ORDER_TRADE_UPDATE`，执行类型 `x: "EXPIRED"`
- 原因码 `gtd`：`4`（被高优先级 RO 订单取消）、`5`（账户清算，保证金不足）
- 订单状态 `X: "EXPIRED"`（终态）

**处理**：
- 标记该批次状态为 `failed`（终态）
- 日志："批次 k @ price 过期未成交（原因码 N）"
- 该批次不产生止盈单，不计入均价
- 不自动补单，用户手动 ← 撤单后 ↑ 补单

### 4.5 EARLY_CLOSE 状态下批次成交

当处于提前平仓状态时，如果价格朝挂单方向移动导致未成交批次被成交：

- **提前平仓单不受影响**（继续挂在盘口）
- 新成交的批次正常挂止盈单
- 重新计算加权均价，更新 SL/SM（≥3 秒节流）
- 状态维持在 EARLY_CLOSE，不自动退出

## 五、平仓场景

### 5.1 止盈触发（单笔）

- 某笔止盈限价单被成交 → 该批次状态变为 `tp_closed`
- 重算剩余活跃批次的加权均价
- 撤销旧的 SL/SM，按新均价重新挂（≥3 秒节流）
- **全部止盈单均已成交 → 轮次结束**：撤销所有剩余挂单（含未成交批次）、止损单、保底单，清理 batch_state

### 5.2 止损触发（统一，依赖币安条件单）

- 统一止损算法单已在币安挂出（STOP_MARKET 类型），触发后币安自动挂 BBO 市价平仓单
- 触发时：**撤销所有剩余未成交开仓挂单**（因为止损价可能比远端挂单价更好，留在盘口无意义）
- **保留保底止损单不动**（SM 不撤）
- 已挂出的止盈限价单由币安自动取消（仓位被止损平掉后，关联的止盈单会被币安撤销，`X: "CANCELED"`）
- 系统通过 WebSocket 感知止损单成交 → 清理本地 batch_state → IDLE

### 5.3 保底止损触发（统一，依赖币安条件单）

- 统一保底止损算法单已在币安挂出（STOP_MARKET 类型），触发后由币安自动市价执行
- 系统通过 WebSocket 感知成交 → 撤销所有挂单、所有止盈单、保底单，清理本地状态
- 处理方式同止损触发

### 5.4 提前平仓（→ 键）

- 挂 1 笔 BBO 限价单，数量 = 当前已成交总量
- 备份所有独立止盈单（tp_backup），撤销止盈单
- 保留统一 SL/SM 不动，保留未成交批次挂单不动
- 进入 EARLY_CLOSE 状态

**撤销提前平仓（← 键）**：
- 撤销提前平仓单
- 从 tp_backup 恢复原来的独立止盈限价单（按原价格、原数量）
- 回到 PARTIAL_FILLED 状态

### 5.5 Z 键市价全平

- 所有已成交批次一次性市价平仓
- 撤销所有挂单、止盈单、止损单、保底单、提前平仓单
- 清理 batch_state → IDLE

## 六、用户交互

### 6.1 撤单（← 键）

**上下文感知**：

| 当前状态 | ← 键行为 |
|---------|---------|
| 有未成交批次 + 无提前平仓 | 一键撤销所有未成交批次挂单。已成交部分和止盈单保留。 |
| EARLY_CLOSE（提前平仓中） | 撤销提前平仓单，恢复备份的止盈单。未成交批次不动。 |
| 全部成交 / 无挂单 | 无效操作 |

### 6.2 补单（新增操作）

**触发条件**：未完全平仓（还有活跃仓位），且满足以下任一：
- 有未成交批次被撤销 → 补回未成交部分
- 有止盈单触发成交 → 补回已止盈部分

**核心逻辑**：只要整体仓位未完全平仓且轮次未结束，始终可以补单。

**补单量计算公式**：

```
补单量 = 最大持仓数量 − 当前未止盈仓位数量 − 当前未成交开仓挂单数量
```

- `最大持仓数量`：目标总名义价值对应的 ETH 数量（恒定，开仓时确定）
- `当前未止盈仓位数量`：所有 `filled` + `tp_placed` 状态批次的 ETH 总和
- `当前未成交开仓挂单数量`：所有 `pending` 状态批次的 ETH 总和

**不用历史已止盈量计算**：因为用户可能来回补仓多次，历史已止盈数量可能超过最大持仓数量。

**行为**：
- 基于**当前最新行情**重新计算阶梯价格
- 挂出新的未成交批次，笔数按补单量比例计算（最少 1 笔）
- 只能补同一方向
- 补单量 ≤ 0 时提示"仓位已满，无需补单"

**示例**：
- 最大持仓 3.3333 ETH，当前未止盈 1.3334 ETH（2 批），挂单中 0.6667 ETH（1 批）→ 补单 = 3.3333 − 1.3334 − 0.6667 = 1.3332 ETH（2 批）
- 最大持仓 3.3333 ETH，当前未止盈 3.3333 ETH（5 批全部成交），挂单中 0 → 补单 = 0，无需补单（全部成交）
- 最大持仓 3.3333 ETH，止盈了 2 批，剩余未止盈 2.0000 ETH，无挂单 → 补单 = 3.3333 − 2.0000 − 0 = 1.3333 ETH（2 批）

**操作方式**：当存在已成交部分时，按方向键自动判断（开仓/补单）

### 6.3 其他按键兼容

| 键 | 分批模式下行为 |
|------|--------------|
| ↑/↓ | 开仓/补单（根据当前状态智能判断） |
| → | 提前平仓（平所有已成交） |
| ← | 上下文感知撤单（见 6.1） |
| Z | 市价全平所有已成交 |
| H | 持仓同步（保持原功能：同步账户余额、刷新历史仓位数据，不合并订单） |
| S | 设置面板 |
| R | 脱敏基数重置（不变） |
| Q | 退出 |

## 七、冲突场景与状态一致性分析

### 7.1 场景一：EARLY_CLOSE + ← 撤单

**初始化**：5 批 LONG，批 1/2 成交 @ 2999/2998，止盈 TP1 @ 3002、TP2 @ 3001；批 3/4/5 挂单中。

**操作序列**：
1. 按 → 提前平仓 → 挂平仓单 @ 3000, 备份 [TP1, TP2], 撤销 TP1/TP2, 进入 EARLY_CLOSE
2. 按 ← → **撤销的是平仓单**，恢复 TP1 @ 3002 和 TP2 @ 3001，回到 PARTIAL_FILLED

**验证**：← 后批 3/4/5 挂单仍在，TP1/TP2 已恢复。

### 7.2 场景二：EARLY_CLOSE + 挂单成交

**初始化**：同 7.1（EARLY_CLOSE 状态，平仓单 @ 3000）。

**操作序列**：
1. 价格下跌到 2997 → 批 3 成交 @ 2997
2. 系统：挂 TP3 @ 3000，重算均价 = (2999+2998+2997)/3 = 2998.00，更新 SL/SM
3. 平仓单仍在 @ 3000，不受批 3 成交影响
4. 状态仍为 EARLY_CLOSE

**验证**：TP3 已挂出，平仓单不变，SL/SM 已更新为新的加权均价。

### 7.3 场景三：全部止盈成交 → 强制结束

**初始化**：3 批 LONG 已成交（批 1/2/3 @ 2999/2998/2997），TP1 @ 3002、TP2 @ 3001、TP3 @ 3000；批 4/5 挂单中。

**操作序列**：
1. 价格上升到 3000 → TP3 成交，批 3 止盈关闭
2. 价格上升到 3001 → TP2 成交，批 2 止盈关闭
3. 价格上升到 3002 → TP1 成交，批 1 止盈关闭
4. 系统检测：已挂出的全部止盈单均成交 → **轮次结束**
5. 撤销批 4/5 挂单、撤销 SL/SM、清理 batch_state → IDLE

**验证**：即使还有未成交的批 4/5，也全部撤销。日志："全部止盈成交，轮次结束"。

### 7.4 场景四：EARLY_CLOSE + 平仓单成交

**初始化**：同 7.1（EARLY_CLOSE 状态，平仓单 @ 3000，2 批已成交）。

**操作序列**：
1. 价格上涨到 3000 → 平仓单成交，批 1/2 全部平仓
2. 系统：撤销批 3/4/5 挂单、撤销 SL/SM、清理 batch_state → IDLE

**验证**：平仓单成交后所有未成交批次一并撤销。

### 7.5 场景五：补单 + 部分止盈振荡

**初始化**：5 批全部成交，TP1-TP5 挂出，全部止盈。

**操作序列**：
1. TP1、TP2 成交 → 剩余 TP3/4/5
2. 按 ↑ 补单 → 挂 2 批新挂单（按当前 BBO 重新计算阶梯）
3. 新批 6/7 成交 → 挂 TP6/TP7
4. 此时有 TP3/4/5/6/7 共 5 个止盈单
5. TP3 成交 → 剩余 TP4/5/6/7
6. 价格下跌触发止损 → 币安 STOP_MARKET 执行，清理一切 → IDLE

**验证**：补单后的新止盈和旧止盈共存，止损触发时全部清理。

### 7.6 场景六：← 撤单后补单 + 价格穿越

**初始化**：5 批部分成交，批 1/2 已成交各有 TP。

**操作序列**：
1. ← 撤销批 3/4/5（未成交）
2. 价格大幅下跌（做多场景），远离原挂单区间
3. ↑ 补单 → 系统用当前 BBO 重新计算阶梯，挂 3 批新单（价格更低）
4. 新批 6 成交 @ 新低价 → 挂 TP6，SL/SM 更新为新加权均价

**验证**：补单价格基于当前行情而非原价，更贴近市场。

## 八、自动化系统适配

### 8.1 移动止损 → 禁用

- 分批模式开启时，移动止损完全不生效
- TUI 账户面板不再展示移动止损网格
- 设置面板中移动止损相关字段自动隐藏（visible_cond 判断 batch_mode.enabled）

### 8.2 浮亏保护 → 适配（分批模式重制）

**触发条件**：开仓后 N 分钟检测浮亏（不变）。

**触发后执行**：

1. **撤销所有未成交开仓挂单**
2. **比较当前价与加权均价**：
   - **当前价 > 加权均价**（做多时价格在均价上方）→ 在加权均价位置挂 `STOP` 条件限价止损单（触发后以均价限价平仓），数量 = 剩余未止盈仓位总量
   - **当前价 ≤ 加权均价**（做多时价格已跌破均价）→ 在加权均价位置挂限价平仓单，数量 = 剩余未止盈仓位总量
3. **禁止后续补单**：设置 `batch_state['supplement_blocked'] = True`，↑↓ 不再提示补单
4. **已挂出的止盈单保留不动**

### 8.3 挂单超时 → 禁用

- 分批模式下不设置挂单超时
- 未成交批次会一直挂在盘口直到手动撤销或市场变动

### 8.4 持仓同步（H 键）→ 保持原功能

- H 键功能不变：同步账户数据、刷新余额、更新历史持仓记录
- 不自动合并订单，不重建批次明细
- 分批模式下，H 键同步时读取交易所当前持仓和挂单状态，与本地 batch_state 做差异对比

## 十二、批量下单接口

### 12.1 接口说明

**`POST /fapi/v1/batchOrders`**（币安合约批量下单）：

| 属性 | 值 |
|------|------|
| 最大笔数 | **5 单/次** |
| timeInForce | 完整支持，含 **GTX**（Post-Only Maker） |
| 原子性 | **非原子**，各订单独立成功/失败 |
| GTX 被拒 | 单笔返回错误码 `-5022`（would-match），其他不受影响 |
| 请求权重 | 5 |
| 其他端点 | `DELETE /fapi/v1/batchOrders`（批量撤单，≤5 单）、`PUT /fapi/v1/batchOrders`（批量修改，≤5 单） |

### 12.2 使用策略

1. **下单**：每 5 笔合并为一次 batchOrders 调用，50 笔 = 10 次调用
2. **Maker 优先**：全部批次以 `timeInForce: "GTX"` 提交；收到 -5022 的批次降级为 GTC 限价单逐笔补挂
3. **回滚策略**：不自动回滚已成功挂出的批次（非原子性，部分成功是正常的）。用户可手动 ← 撤单
4. **撤单**：← 键使用 `DELETE /fapi/v1/batchOrders`（每 5 单一组）撤销所有未成交批次

## 十三、技术风险与对策

1. **并发成交**：WebSocket 可能在极短时间内推送多笔成交。对策：3 秒合并窗口，只触发最后一次 SL/SM 重挂。
2. **Algo Order 速率限制**：每笔成交后都要撤销 + 重挂 SL/SM（2 次 API 调用）。对策：≥3 秒节流 + 合并窗口，将同一窗口内的多次成交合并为一次 SL/SM 更新。
3. **补单价格重算**：补单时行情可能已大幅变动，阶梯价格需要基于最新订单簿重新计算。
4. **API 错误处理**：批量下单非原子，部分成功是常态。不自动回滚，用户可手动撤单。
5. **TUI 空间**：止盈单和挂单各最多显示 5 条，按价格接近当前价排序，超出部分折叠。
6. **Maker 优先**：使用 GTX（Post-Only）下单，被拒后自动降级为 GTC 限价单补挂。
7. **状态一致性**：EARLY_CLOSE 状态下挂单成交、← 撤单语义等已在第七章详细分析。

## 十四、WebSocket 可靠性修复与 REST 兜底

### 14.1 现有问题诊断

| 问题 | 位置 | 影响 |
|------|------|------|
| **应用层心跳缺失** | `user_stream_ws.py:98` — `asyncio.TimeoutError` 仅记录日志，不重连 | 僵尸连接永久沉默，表面连接正常但无数据流入 |
| **Listen Key 过期不轮换** | `user_stream_ws.py:122-125` — 重连用旧 key，过期后死循环 | 60 分钟后 WS 彻底失效 |
| **无消息超时无保护** | 两个 WS 均无"上次收到消息"时间戳检测 | 无法区分"正常空闲"和"连接已死" |
| **订单回调同步执行** | `user_stream_ws.py:155` — `_on_order_update` 是同步函数 | 回调执行阻塞事件循环，高并发时积压 |

### 14.2 修复方案

#### 14.2.1 应用层心跳与超时检测

```
市场数据 WS（BinanceListener）:
  - 已有：30s recv 超时 → 重连（工作正常，bookTicker 每 200ms 推送）
  - 保持现状

用户数据流 WS（UserStreamWebSocket）:
  - 新增：上次消息时间戳 self._last_msg_ts
  - 新增：每 30s 检查 (now - _last_msg_ts) > 60s → 强制断开重连
  - 新增：重连时自动获取新 listenKey（不再复用旧 key）
  - 修复：asyncio.TimeoutError → 触发重连（不再仅记录日志）
```

#### 14.2.2 Listen Key 轮换

```python
# user_stream_ws.py 重连流程：
# 旧流程：用旧 listen_key 直连 → 过期后死循环
# 新流程：
#   1. 检测到需要重连
#   2. 调用 api_client.get_listen_key() 获取新 key
#   3. 用新 key 建立 WebSocket
#   4. 旧 key 通过 api_client.close_listen_key(old_key) 清理
```

#### 14.2.3 连接健康状态

在 `batch_state` / `LiveTrader` 中维护：

```python
self._ws_health = {
    'user_stream_connected': False,     # WS 连接状态
    'user_stream_last_msg_ts': 0.0,     # 上次收消息时间
    'user_stream_msg_count': 0,         # 累计消息数
    'user_stream_restart_count': 0,     # 重连次数
    'fallback_active': False,           # 是否已切换到 REST 兜底
}
```

TUI 账户面板右上角显示 WS 状态指示灯：
- 🟢 WS 正常（60s 内有消息）
- 🟡 WS 空闲（60s~120s 无消息）
- 🔴 WS 断开 / REST 兜底中

### 14.3 REST 兜底轮询

当 WS 不可用时（`fallback_active = True`），启动 3 秒间隔的 REST 轮询：

| 轮询项 | REST 接口 | 频率 | 用途 |
|--------|-----------|------|------|
| 未成交挂单 | `GET /fapi/v1/openOrders` | 3s | 检测批次/止盈/平仓单状态变化 |
| 持仓 | `GET /fapi/v2/positionRisk` | 3s | 检测仓位变化（止盈/止损成交） |
| 账户余额 | `GET /fapi/v2/account` | 3s | 同步余额 |
| 最近成交 | `GET /fapi/v1/userTrades` | 3s | 补充检测成交（差额比对） |

**性能分析**：

```
所有 REST 调用沿用现有模式：await asyncio.to_thread(self.api.xxx, ...)
→ 同步 requests 在线程池执行，不阻塞 asyncio 事件循环
→ TUI 刷新不受影响

单次轮询权重：5 + 5 + 5 + 5 = 20
每分钟权重消耗：20 × 20 = 400
币安限制：1200 权重/分钟
占用比例：33%

仅在 fallback_active=True 时消耗，WS 正常时权重消耗为 0
```

**切换逻辑**：
1. WS 60s 无消息 → `fallback_active = True`，启动轮询
2. WS 恢复收到消息 → 对比本地状态与 WS 事件，对齐后 `fallback_active = False`，停止轮询
3. TUI 日志提示："WebSocket 异常，已切换 REST 轮询模式" / "WebSocket 已恢复"

### 14.4 事件输入抽象

核心处理逻辑不直接依赖 WS 或 REST，而是通过统一的事件入口：

```python
def _on_batch_event(self, event_type: str, payload: dict, source: str = 'WS'):
    """
    event_type: 'BATCH_FILLED' | 'TP_FILLED' | 'SL_FILLED' | 'SM_FILLED' | 
                'ORDER_EXPIRED' | 'ORDER_CANCELED' | 'BALANCE_CHANGE'
    source: 'WS' | 'REST_POLL'
    """
```

- `source='WS'`：由 `_on_order_update` / `_on_account_update` 调用
- `source='REST_POLL'`：由 REST 轮询比对 diff 后调用
- 同一事件从两个来源到达时，以 `orderId` + `status` 做幂等去重

### 14.5 清理修复

- `cleanup()` 中调用 `api_client.close_listen_key(listen_key)` 销毁 listen key（当前缺失）
- 退出时确保两个 WS 任务正确取消，避免资源泄漏

## 十八、v1.9.0 实现订正（2026-06-30）

以下为 v1.9.0 上线后发现并修复的实现级 bug，订正至本文档以保持设计与代码一致。

### 18.1 双向持仓模式 `reduceOnly` 参数误传

**问题**：`place_order()` 和 `place_algo_order()` 均不接受 `reduceOnly` 参数（双向持仓模式不允许此参数）。分批模式 6 处代码传入了 `reduceOnly=True`，导致 `TypeError`，被各处的 `except Exception: pass` 静默吞掉。

**影响**：
- 止盈限价单（GTX + GTC 降级）全部静默失败
- 保底止损市价单（STOP_MARKET）静默失败
- 提前平仓单静默失败
- 提前平仓撤销后的止盈恢复静默失败
- 浮亏保护限价平仓单静默失败

**修复**：`src/trading/live.py` — 移除 `_place_batch_tp`、`_update_batch_sl_sm`、`_batch_early_close`、`_cancel_batch_early_close`、`_place_loss_protection_limit` 中所有 `reduceOnly=True` 传参（共 6 处）。

### 18.2 `float(None)` 状态推送崩溃

**问题**：`server.py` `_build_state()` 中 `b.get('tp_price', 0)` 在 key 存在但值为 `None` 时不走默认值（Python `.get()` 只在 key 缺失时用默认值），导致 `float(None)` 抛出 `TypeError`，状态推送循环每秒钟崩溃一次。

**修复**：`src/web/server.py:395` — 改为 `b.get('tp_price') or 0`。同时异常处理中增加 `logs/_crash_state.log` 完整堆栈写入。

### 18.3 保底止损参数名错误

**问题**：`_update_batch_sl_sm()` 中 `place_algo_order(**{stopPrice=...})` 传参名为 `stopPrice`，但 `place_algo_order`（Algo Order API `/fapi/v1/algoOrder`）的参数名为 `triggerPrice`。`stopPrice` 是普通订单 API 的参数名。导致 `TypeError` 被 `except Exception` 静默吞掉。

**附带差异**：分批模式 `workingType='CONTRACT_PRICE'`，非分批模式使用 `MARK_PRICE`，订正为一致使用 `MARK_PRICE`。

**修复**：`src/trading/live.py` `_update_batch_sl_sm` — `stopPrice` → `triggerPrice`，`workingType` → `MARK_PRICE`，增加成功/失败 INFO 日志。

### 18.4 SL/SM 更新节流丢失

**问题**：`_schedule_sl_sm_update()` 的 3 秒节流逻辑在命中节流时设置 `_pending_sl_update = True`，但该标志无任何代码检查。若第二笔成交在 3 秒内到达，SL/SM 更新被永久丢弃。

**修复**：`src/main_live.py` 价格 tick 循环中增加兜底检查，当 `_pending_sl_update == True` 且距上次更新 ≥3 秒时，自动触发 `_schedule_sl_sm_update()`。

### 18.5 浮亏保护 UI 显示 undefined

**问题**：分批模式下 `loss_protection_manager.set_entry_info()` 从未调用（仅在非分批模式 `_safe_place_tp_sl_orders` 中调用）。`get_status()` 只返回 `{status:'等待开仓'}`，缺少 `remaining_time` 和 `pnl_status` 字段。前端渲染为 "浮亏保护 undefined (undefined)"。

**修复**：
- `src/trading/live.py` `_handle_batch_fill` — 首笔批次成交时调用 `loss_protection_manager.set_entry_info()`
- `src/trading/live.py` `_init_batch_state` — 重置 `_max_float_pnl` / `_min_float_pnl` 防止上轮数据泄漏

### 18.6 分批模式持仓卡片数据缺失

**问题**：`server.py` 构建分批模式 position 时：
- 缺少 `open_time` → 前端倒计时定时器不启动
- 从 `batch_state['peak_floating_profit']` 读取浮盈峰值，但该 key 从未被写入（`update_price` 写入的是 `_max_float_pnl` 属性）→ 最高浮盈浮亏始终为 0

**修复**：`src/web/server.py` — 分批模式 position 改为从 `_max_float_pnl` / `_min_float_pnl` 读取（与非分批一致），补齐 `_max_float_pnl_price` / `_min_float_pnl_price`，取首笔批次 `fill_time` 作为 `open_time`。

### 18.7 REST 兜底不检测止盈成交

**问题**：`_rest_fallback_loop` 对所有成交流水无条件分发 `BATCH_FILLED`，不区分开仓成交/止盈成交，导致止盈成交不被检测。同时只检查 `status=='pending'` 批次的订单消失，不检查 `tp_placed` 批次。UserStream WS 零消息时，止盈成交永远不被感知 → 系统认为持仓仍在。

**附带**：`_check_disappeared_order` 中 `self.api.get_order(self.symbol, orderId=order_id)` 参数名错误（应为 `order_id`），被 `except Exception: pass` 静默吞掉。

**修复**：
- `src/trading/live.py` `_rest_fallback_loop` — 成交流水按 `orderId` 匹配批次 `order_id` 或 `tp_order_id` 后分发正确事件类型
- 新增 `_check_disappeared_tp_order` 方法，对 `tp_placed` 批次检测订单消失
- 修正 `_check_disappeared_order` 中 `get_order` 参数名 `orderId` → `order_id`

### 18.8 TP 成交检测盲区（独立监控 → 全局批量轮询）

**问题**：18.7 修复了 REST 兜底循环的检测逻辑，但兜底循环仅在 `fallback_active=True` 时运行（WS 超 60s 无消息才激活）。若 WS 每隔几十秒收到一条 ACCOUNT_UPDATE 消息，`last_msg_ts` 持续刷新，兜底永不激活，止盈成交仍不被检测。

**v1 修复**（已废弃）：新增 `_monitor_batch_tp_order`，每笔止盈单独立 `get_order` 轮询。200 笔时 2400 权重/min。

**v2 修复**（当前）：替换为全局批量监控 `_batch_tp_monitor_loop`：
- 全局唯一任务，`_start_batch_tp_monitor` 幂等启动，`_stop_batch_tp_monitor` 在 `_cleanup_batch_state` 停止
- 每 5s 调一次 `get_open_orders`（权重固定 5），O(1) 不随订单数增长
- 订单消失（不在 open_orders 中）→ 单独 `get_order` 确认 FILLED 或 CANCELED
- 200 笔 TP 单只需 60 权重/min，远低于 2400 限额
- `_place_batch_tp` 和 `_update_batch_sl_sm` 补充 `_rebuild_key_prices()` 调用，确保 key price 穿透检测及时更新

### 18.9 补单重复提交 + 前端方向按钮 + `max_position_size` 累加

**问题 1**：`_place_batch_orders` 遍历全部 `bs['batches']` 提交 batchOrders 而不是只提交新增批次，导致补单时重复提交已成交/已取消/已止盈的批次。

**问题 2**：`_supplement_batch_orders` 调用 `_place_batch_orders` 后，`_place_batch_orders` 内 `bs['batches'] = [b for b in batches if b.get('order_id')]` 会丢弃已成交批次，再重置 `total_count`，计数混乱。

**问题 3**：分批模式有持仓时，前端未禁止反向开仓按钮。

**问题 4**：`max_position_size` 在每次 `_place_batch_orders` 成功后直接赋值为已挂批次总和，补单时被覆盖为较小值。

**修复**：
- `_place_batch_orders`：只提交 `status=='pending' and not b.get('order_id')` 的新批次；不再过滤 `bs['batches']` 列表、不再修改 `total_count`
- `_supplement_batch_orders`：移除 `bs['total_count'] = len(bs['batches'])` 赋值
- `max_position_size` 改为累加（`+=`）而非替换（`=`）
- 前端 `updatePosCard`：分批模式有持仓时 `dirSide` 取 `pos.side`，LONG 禁用 SHORT 按钮，SHORT 禁用 LONG 按钮
